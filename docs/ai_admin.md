Спланируй новый модуль - ИИ-агента админисратора БД. Он должен иметь в качестве инструкментов доступ ко всем моделям БД, а также хорошо понимать, какая модель за что отвечает. Он также должен иметь средства для поиска информации. Изучи уже реализованного агента, нового будем делать на том же стеке. 
Новый агент должен запускаться в celery, управляться асинхронными задачами. В один момент времени может выполняться только одна задача. (на постановку новых должны бросать ошибку). 
Резульат задачи должен состоять из двух компонентов: изменения в моделях, сделанные и проверенные в процессе решения задачи (если требуется), и сообщение для пользователя (должно включать отчет об изменениях, если они выполнялись). 

Примерная архитектура самого агента - llm workflow + ai agent, т.е.:
модуль анализа задачи, который определяет, понятна ли задача и достаточно ли в ней данных. Если нет, он должен запросить дополнительную информацию, завершив задачу с сообщением-вопросом. Если задача понятна, он должен запустить второй модуль -  ai agent, который будет выполнять задачу. Это агент должен получить от модуля анализа хорощшо сформулированную задачу. Этот агент уже должен иметь перечисленные выше инструменты. Его задача - сделать изменения в моделях, решающие поставленную задачу. Изменения должны протоколироваться тем или иным способом (продумай это). Когда агент завершил работу, контроль должен передаваться к модулю анализа, который проанализирует протокол работы агента и сравнит с поставленной задачей. Этот агент должен принять решение: передать контроль модулю описания для составления отчета для пользователя, или снова передать контроль агенту , дополнив задачу списком доработок, которые осталось сделать. Выбор этого модуля  также должен протоколироваться.  Когда контроль окажется у модуля описания, он долежн собрать html отчет по протоколу работы агента, указав в нем все необходимое в красивой форме (у него должен быть подготовленный шаблон). Затем он должен передать контроль модулю ответа, который соберет сообщение для пользователя. В результат задачи должно быть записано: протокол работы, ссылка на отчет (это должна быть отдельная ручка, сам отчет хранится в базе), ответ для пользователя. 
Мысли по реализации:
1) Каждый модуль описывается как ИИ-агент на том же фреймворке, которым мы уже пользуемся. 
2) Передача контроля реализована как запуск задачи в celery. Т.е. отдельная задача в celery для каждого модуля.
3) Имеем следующих агентов в системе:
    1) анализа задачи 
    2) ИИ-администратор
    3) Контролирующий агент
    4) Описателя
    5) Ответчика
4) Подробности по агенту анализа задачи:
    1) имеет доступ к истории сообщений с пользователем по принйципу пагинации (сразу последние N сообщений, остальные по запросу из тулзы, страницами по N)
    2) имеет доступ к списку задач, в виде индекса по именам. Может тулзой по любой дернуть задачу, чтобы посмотреть детали: протокол, отчет, ответ (отдельными запросами, не все сразу)
    3) Может завершать задачу досрочно, без выполнения агента ИИ-администратора и прочих, такие задачи помечаются как "dialog_only", в таком случае этот агент может сам формировать сообщение для пользователя. Задача, соответственно, не имеет ни отчета, ни протокола. Этот агент должен пользоваться короткой формой завершения задачи, если не ясна задача от пользователя, либо пользователь просто что-то уточняет, ничего делать с моделями\БД не нужно. 
5) Подробности по агенту ИИ-администратора:
    1) Ничего не знает про историю общения с пользователем. Знает только поставленную предыдущим агентом задачу и протокол ее решения, в случае 2 и последующих запусков на этой задаче. 
    2) имеет описание всех моделей, имеет тулзы для получения объектов этих моделей, создания новых
    3) имеет тулзы для поиска в сети
    4) Операции с моделями реализуй  как набор тулзов (их описание должно иметь примеры)
    5) Может принять решение о том, что решение задачи невозможно. В этом случае он должен назвать одну из двух причин и оставить комментарий. Возможные причины: 1) Требуется модификация структуры БД;  2) Концептуально нерешаемо
6) Подробности по агенту контроля:
    1) Ничего не знает про историю общения с пользователем. Знает только поставленную предыдущим агентом задачу и протокол ее решения.
    2) Может либо вынести вердикт "задача решена" (тогда контроль уйдет к описателю), либо вернуть задачу обратно в очередь для ИИ-администратора, составив список недоработок, которые надо исправить.
7) Подробности по агенту описателя:
    1) Ничего не знает про историю общения с пользователем. Знает только поставленную предыдущим агентом задачу и протокол ее решения.
    2) имеет подготовленный HTML-шаблон в контексте, в котором используется html + bootstrap + css\js
    3) Отчет должен содержать задачу, протокол ее решения в удобной форме, обязательно разбитый на запуски ИИ-администратора
    4) Отчет сохраняется в отдельную таблицу, просмотреть можно по отдельной ручке, в идеале не по ID, а по slug, slug можно составить как task_name_report_date_time
8) Подробности по агенту ответчика:
    1) имеет доступ к истории сообщений с пользователем по принйципу пагинации (сразу последние N сообщений, остальные по запросу из тулзы, страницами по N)
    3) имеет в контексте постановку текущей задачи, ее протокол решения.
    4) Должен составить ответ для пользователя, его текстовую часть. Ответ должен содержать краткое резюме по решению задачи. 
    
Ручки для доступа к агентам должны защищаться авторизацией суперпользователя или предъявлением специального апи-ключа.
